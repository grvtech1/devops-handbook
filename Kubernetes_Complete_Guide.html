00

Cluster ka anatomy — sabse pehle yeh

Baaki sab objects isi cluster ke andar rehte hain. Cluster ke do hisse: Control Plane (dimaag — decisions leta hai) aur Worker Nodes (haath-pair — actual containers yahan chalte hain).

🧠 Control Plane  ·  decisions
API Server
Har command yahan aati hai. Cluster ka front door — kubectl isi se baat karta hai.
etcd
Cluster ki poori state ka database. "Kya hona chahiye" yahan likha hai.
Scheduler
Naya Pod kis Node pe chalega — yeh decide karta hai.
Controller Mgr
"Actual = desired?" Bar-bar check karta hai, mismatch fix karta hai.
💪 Worker Nodes  ·  yahan containers chalte hain
kubelet
Har node ka agent. API Server se order leta hai, containers chalata hai.
containerd
Actual container runtime. Image pull karke container start karta hai.
kube-proxy
Service ka traffic sahi Pod tak — iptables rules banata hai.
Calico (CNI)
Pod ko IP deta hai + node-to-node routing.
◆
Ek line mein: Tum bolte ho "3 frontend chahiye" → API Server etcd mein likhta hai → Scheduler nodes choose karta hai → kubelet containerd se containers chalata hai → Controller dhyan rakhta hai ki hamesha 3 hi rahein.
ℹ
VANTA ek kind cluster hai (control-plane + worker ek hi Docker container mein simulate). Billfree mein Terraform AWS pe alag EC2 machines banata hai, phir kubeadm unhe real control-plane + worker nodes bana deta hai.
01

Workloads — jo actually app chalate hain

Yeh objects tumhare containers ko chalate aur zinda rakhte hain. Sabse chhoti unit Pod hai; baaki sab Pod ko manage karne ke tareeke hain.

📦
Pod
kind: Pod

Kubernetes ki sabse chhoti unit. Ek Pod ke andar ek (ya kabhi zyada) container hota hai. Kubernetes container ko directly nahi, hamesha Pod ke through chalata hai.

◆
Pod = container + uska IP + storage + config, sab ek lifafe mein. Pod ephemeral hai — mar gaya to naya banta hai, purana wapas nahi aata (naya IP milta hai).
VANTA

frontend Pod — Go ka web server, port 8080. Delete kar do to Deployment turant naya bana deta hai.

BILLFREE

auth-service Pod — login/token handle karta hai, port 8080. HPA load pe 2 se 6 tak Pods banata hai.

⚠
Direct Pod kabhi nahi banate production mein. Mar gaya to koi wapas nahi laata. Isliye hamesha Deployment (ya StatefulSet/DaemonSet) use karte hain — wo Pod ko zinda rakhta hai.
👥
ReplicaSet
kind: ReplicaSet

"Hamesha N copies (replicas) chalti rahni chahiye" — yeh guarantee ReplicaSet deta hai. Ek Pod mara → ReplicaSet turant naya bana deta hai.

◆
Tum ReplicaSet khud nahi likhte — Deployment tumhare liye automatically banata hai. Labels se apne Pods pehchaanta hai.
ReplicaSet (app=frontend, replicas=3) ├── Pod frontend-a1b2 ✓ Running ├── Pod frontend-c3d4 ✓ Running └── Pod frontend-e5f6 ✗ mar gaya → naya banaya → frontend-g7h8 ✓
ℹ
Dono projects mein tum kubectl get rs chalao to har service ka ReplicaSet dikhega — par tumne kabhi ReplicaSet YAML nahi likhi. Deployment ne banaya.
🚀
Deployment
kind: Deployment

Sabse zyada use hone wala object. Stateless apps ke liye. Yeh ReplicaSet manage karta hai, aur uske through Pods. Iska superpower: rolling update aur rollback.

◆
Naya image deploy karo → Deployment ek-ek karke purane Pod hataata hai, naye laata hai — zero downtime. Kuch galat hua? kubectl rollout undo se wapas.
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
  labels: { app: frontend }   # (1) Deployment ka label
spec:
  replicas: 3
  selector:
    matchLabels: { app: frontend }  # (2) "yeh Pods mere"
  template:                       # (3) Pod ka blueprint
    metadata:
      labels: { app: frontend }  # (4) Pod pe yeh label (2 se match!)
    spec:
      containers:
        - name: server
          image: docker.io/grvp1/frontend:latest
VANTA

12 microservices — sab Deployments. frontend, cartservice, checkoutservice… Dev mein 1 replica, Prod overlay mein 3.

BILLFREE

api-gateway, auth-service, report-service… sab reusable Helm chart se Deployment bante hain, default 2 replicas.

🗄️
StatefulSet
kind: StatefulSet

Deployment ka bhai, par stateful apps ke liye — jinke paas apna data hai (databases). Har Pod ko fixed naam + apna storage milta hai jo restart pe bhi rehta hai.

◆
Deployment: Pods interchangeable, koi bhi chalega (frontend-x1, frontend-x2 — farak nahi). StatefulSet: postgres-0 hamesha postgres-0 rahega, apne 5Gi disk ke saath.
kind: StatefulSet
metadata: { name: postgres, namespace: billfree }
spec:
  serviceName: postgres
  replicas: 1
  template: { ... postgres:16-alpine container ... }
  volumeClaimTemplates:            # har Pod ko apna PVC
    - metadata: { name: data }
      spec:
        accessModes: [ReadWriteOnce]
        resources: { requests: { storage: 5Gi } }
VANTA

redis-cart cart data rakhta hai. Reviews ke liye optional postgres component bhi StatefulSet-style persistence use karta hai.

BILLFREE

postgres — main database, StatefulSet with 5Gi PVC via volumeClaimTemplates. Pod restart → wahi data wapas.

🛰️
DaemonSet
kind: DaemonSet

"Har Node pe exactly ek Pod chalao" — yeh DaemonSet karta hai. Naya node juda? Automatically wahan bhi ek Pod. Node hata? Pod bhi gaya.

◆
Yeh un cheezon ke liye hai jinhe har machine pe chahiye: networking agent, log collector, metrics exporter.
VANTA

calico-node — har node pe 1 Pod, pod-IP aur routing sambhalta hai. kube-proxy bhi DaemonSet.

BILLFREE

Same Calico + kube-proxy. Plus node-exporter (Prometheus ke liye) — har node ka CPU/RAM/disk metrics deta hai.

⚙️
Job & CronJob
kind: Job / CronJob

Deployment "hamesha chalta rehta hai". Job ek kaam karke khatam ho jaata hai (exit → done). CronJob = Job jo schedule pe chalta hai (jaise cron).

VANTA

loadgenerator traffic banata rehta hai (yeh Deployment hai). Ek-baar-ke kaam ke liye Job pattern use hota.

BILLFREE

migrate-job — deploy se pehle database migrations chalata hai (tables banata/update karta), phir khatam. Classic Job.

✓
Kab kya: Hamesha chalna hai → Deployment. Data + identity chahiye → StatefulSet. Har node pe → DaemonSet. Ek baar chal ke khatam → Job. Schedule pe → CronJob.
02

Networking — jo sab ko jodta hai

Pods to ban gaye, par ek-doosre se aur bahar-duniya se baat kaise karein? Yeh part wahi hai — labels se lekar Calico tak.

🏷️
Labels & Selectors
glue of k8s

Labels = Pods pe chipke stickers (app: frontend). Selectors = "mujhe app=frontend wale Pods do". Isi ek mechanism pe poora Kubernetes khada hai.

◆
Deployment apne Pods labels se pehchaanta hai. Service traffic labels se route karti hai. HPA labels se scale karti hai. Sab jagah wahi label.
Deployment selector: app=frontend ──┐ ├──► dhundhta hai Pod labels: app=frontend ◄─────────┘ Service selector: app=frontend ──► traffic isi Pod ko
VANTA

Simple style: app: frontend, app: cartservice. Seedha aur saaf.

BILLFREE

Production standard labels: app.kubernetes.io/name: postgres, app.kubernetes.io/part-of: billfree-techops. Zyada descriptive, tooling-friendly.

⚠
Deployment ka selector.matchLabels, Pod template ka labels, aur Service ka selector — teeno match hone chahiye. Ek bhi typo → Service ka traffic kisi Pod tak nahi pahunchega.
🔌
Service
kind: Service

Pods ke IP badalte rehte hain (mar-mar ke naye). Service ek fixed naam + fixed virtual IP deti hai jiske peeche Pods aate-jaate rehte hain. Plus load balancing free.

◆
Tum cartservice:7070 pe call karo — Service peeche jitne bhi cartservice Pods hain unmein baant deti hai. Pod mara/naya bana, tumhe farak nahi padta.

4 types — access ka daayra:

  • ClusterIP (default) — sirf cluster ke andar se access. Internal service-to-service. Zyadatar sab.
  • NodePort — node ke IP pe ek port khol deta hai (30000+). Bahar se access, par kaccha tareeka.
  • LoadBalancer — cloud ka real load balancer deta hai (AWS ELB). Public internet-facing.
  • Headless (clusterIP: None) — koi virtual IP nahi, seedha Pod IPs. StatefulSet/DB ke liye.
VANTA

frontend internal = ClusterIP. frontend-external browser ke liye — base mein LoadBalancer, par kind pe NodePort 30080 mein patch (kyunki local pe real LB nahi).

BILLFREE

Sab services ClusterIP (internal). Bahar ki duniya Ingress se aati hai, seedha Service se nahi. postgres Service DB tak internal reach deti hai.

🚪
Ingress
kind: Ingress

Cluster ka smart main gate. Ek hi entry point, aur URL/host dekh ke sahi Service pe bhejta hai. TLS (HTTPS) bhi yahin terminate hota hai.

◆
Har service ke liye alag LoadBalancer mehnga aur ganda. Ek Ingress lagao → /api → api-gateway, /auth → auth-service, app.com → web. Ek gate, smart routing.
Internet │ ▼ [ Ingress Controller (nginx) ] ← ek entry, TLS yahan ├── billfree.com/api → api-gateway Service ├── billfree.com/auth → auth-service Service └── billfree.com/ → web Service
ℹ
Ingress sirf rules (YAML) hai. Use lagoo karne wala actual software = Ingress Controller (jaise ingress-nginx). Bina controller ke Ingress kuch nahi karta.
VANTA

Local kind pe Ingress use nahi kiya — seedha NodePort se localhost:8088. Learning ke liye simple rakha.

BILLFREE

ingress-nginx ArgoCD se deploy. Helm chart mein har service ka ingress.enabled + host/path. Production ka asli main gate.

📖
CoreDNS & kube-proxy
cluster plumbing

Do behind-the-scenes heroes jo Service ko kaam karwaate hain.

  • CoreDNS — cluster ka phonebook. cartservice naam ko Service ke virtual IP 10.96.45.12 mein badalta hai. Isliye tum code mein IP nahi, naam likhte ho.
  • kube-proxy — har node pe iptables rules banata hai. Service IP pe aaya packet → assli Pod IP pe redirect. Load balancing yahi karta hai.
frontend: dial("cartservice:7070") │ CoreDNS: cartservice → 10.96.45.12 (Service IP) ▼ kube-proxy: 10.96.45.12 → 10.244.2.8 (asli Pod IP) cartservice Pod ✓
✓
Dono projects mein yeh kube-system namespace mein already chal rahe hote hain — tumhe kuch setup nahi karna. kubectl get pods -n kube-system se dikhte hain.
🕸️
CNI / Calico
DaemonSet · kube-system

CNI = Container Network Interface. Calico ek popular CNI hai. Iske do kaam: (1) har Pod ko IP dena, (2) alag-alag nodes ke Pods ko packet route karna.

◆
Bina CNI ke Pod ka koi IP nahi hota, koi network nahi. Calico har node pe (DaemonSet) baithta hai aur ek virtual network banata hai jahan har Pod ka apna IP hai.
Node 1 Node 2 frontend 10.244.1.5 cartservice 10.244.2.8 │ Calico agent Calico agent ▲ └──────── physical network ─────────────────┘ Calico routing: "10.244.2.x Node 2 pe hai → wahan bhejo"
ℹ
Calico vs Service: Calico packet ko node tak pahunchata hai (IP routing). Service packet ko sahi Pod tak (load balance). Calico road banata hai, Service traffic director hai.
VANTA

kind apna default CNI (kindnet) laata hai — simple. Chhote setup ke liye kaafi.

BILLFREE

kubeadm cluster mein Calico install hota hai — real pod networking + NetworkPolicy support ke saath.

🛡️
NetworkPolicy
kind: NetworkPolicy

Default mein har Pod har Pod se baat kar sakti hai — koi rok nahi. NetworkPolicy = firewall rules: "kaun kis se baat kar sakta hai".

◆
Security ke liye zaroori. Jaise: "database Pod se sirf backend baat kare, frontend ya bahar se koi seedha DB touch na kare."
kind: NetworkPolicy
spec:
  podSelector: { matchLabels: { app: postgres } }
  ingress:                       # postgres mein andar kaun aa sakta
    - from:
      - podSelector: { matchLabels: { app: auth-service } }
      # sirf auth-service → postgres. Baaki sab BLOCKED.
VANTA

Learning setup — NetworkPolicy nahi lagayi (sab khula, simplicity ke liye).

BILLFREE

deploy/platform/networkpolicies.yaml — production mein DB aur services ke beech proper isolation. Calico enforce karta hai.

03

Config & Storage

App ko settings, secrets aur permanent data chahiye. Yeh objects wahi dete hain — bina image ko chhue.

🗂️
Namespace
kind: Namespace

Cluster ke andar logical folders. Same naam ki cheezein alag namespaces mein rakhi ja sakti hain. Isolation + resource limits + access control ke liye.

◆
Namespace = folder, cluster = hard disk. Chhoti app: default namespace kaafi. Badi/multi-team: alag namespaces zaroori.
VANTA

Sab kuch default namespace mein — ek app, ek developer, simple. System cheezein kube-system mein.

BILLFREE

App billfree namespace mein, monitoring alag monitoring namespace mein, ingress-nginx apne namespace mein. Proper separation.

✓
Interview line: "Namespace cluster ko logically partition karta hai — mainly multi-team/multi-environment ke liye. Single app ke liye default namespace sufficient hai."
📝
ConfigMap
kind: ConfigMap

App ki non-secret settings — env vars, config files, URLs. Image ke bahar rakhi jaati hain taaki config badalne pe image rebuild na karna pade.

◆
"12-factor app" ka rule: config code se alag. Same image dev/staging/prod mein chale, bas ConfigMap alag.
VANTA

frontend ko sab service addresses env vars se milte hain: CART_SERVICE_ADDR=cartservice:7070 — Deployment ke env: mein.

BILLFREE

Helm chart mein env / envFrom — non-secret settings ConfigMap-style inject hote hain per environment.

🔐
Secret
kind: Secret

ConfigMap jaisa hi, par sensitive data ke liye — DB passwords, API keys, tokens. Base64 encoded (encryption nahi!), access RBAC se controlled.

⚠
Golden rule: Secret ki asli value kabhi Git mein commit mat karo. Git mein daala = hamesha ke liye history mein, sabko dikhega.
VANTA

kubeconfig, tfstate, tfvars — sab .gitignore mein. Koi secret repo mein nahi.

BILLFREE

postgres-secret Git mein nahi — bootstrap script randomly generate karke banata hai. StatefulSet use karta hai envFrom: secretRef. Real setup mein SealedSecrets/External Secrets.

💾
PV · PVC · StorageClass
persistent storage

Pod ephemeral hai — data uske saath mar jaata hai. Permanent data ke liye yeh teen: storage maang, storage supply, aur auto-provisioning.

  • PVC (PersistentVolumeClaim) — "mujhe 5Gi disk chahiye" — Pod ki maang.
  • PV (PersistentVolume) — asli storage piece jo maang poori karta hai — supply.
  • StorageClass — "aap khud bana lo" — cloud se automatic disk provision (AWS EBS volume).
◆
PVC restaurant ka order hai, PV wo dish, StorageClass wo kitchen jo on-demand bana deti hai. Pod PVC maangta hai, cloud automatically EBS disk deta hai.
VANTA

Local kind — redis-cart emptyDir/local storage. Learning ke liye persistence halka rakha.

BILLFREE

postgres StatefulSet ka volumeClaimTemplates → 5Gi PVC → AWS StorageClass se real EBS volume. Pod restart pe wahi disk wapas.

04

Reliability & Scaling — production ki jaan

App chal jaana kaafi nahi. Load pe scale ho, crash pe khud theek ho, maintenance pe down na jaaye — yeh objects wahi ensure karte hain.

🩺
Health Probes
liveness · readiness · startup

Kubernetes ko kaise pata Pod healthy hai? Probes se — regular health check-ups.

  • livenessProbe — "zinda hai?" Fail → Pod restart. (App hang ho gaya to.)
  • readinessProbe — "traffic lene ke layak hai?" Fail → Service se hata do (mara nahi, bas traffic nahi). (Abhi startup ho raha, ya busy.)
  • startupProbe — dheere start hone wale apps ke liye — "pehle poora start hone do, tab baaki probes shuru."
readinessProbe:
  httpGet: { path: /readyz, port: 8080 }
livenessProbe:
  httpGet: { path: /healthz, port: 8080 }
  initialDelaySeconds: 10   # itna wait, phir check
VANTA

frontend /_healthz pe probe. Yaad hai — emailservice/recommendationservice kind pe CrashLoop kar rahe the kyunki liveness timeout bahut aggressive tha (1s). Relax kiya to fix.

BILLFREE

Har service /healthz (liveness) + /readyz (readiness). postgres pg_isready exec probe se check hota hai.

⚖️
Requests & Limits
resources

Har container batata hai use kitna CPU/RAM chahiye. requests = minimum guarantee (Scheduler isse dekh ke node choose karta hai). limits = maximum, isse zyada nahi.

◆
requests hotel booking hai (yeh to milega hi). limits ceiling hai (isse upar gaye to — CPU throttle, RAM cross to OOMKilled).
resources:
  requests: { cpu: 50m, memory: 96Mi }   # guarantee
  limits:   { cpu: 500m, memory: 256Mi }  # ceiling
ℹ
500m = 0.5 CPU core (m = millicores). 256Mi = 256 mebibytes RAM. Exit code 137 jo VANTA mein aaya tha = OOMKilled ya liveness ne SIGKILL kiya.
VANTA

frontend: requests 100m/64Mi, limits 200m/128Mi. Halka, kind ke chhote resources ke liye.

BILLFREE

Chart default: requests 50m/96Mi, limits 500m/256Mi. postgres ko zyada: limit 1 CPU/512Mi.

📈
HPA — Horizontal Pod Autoscaler
kind: HorizontalPodAutoscaler

Load badhe → Pods automatically badhao. Load kam → Pods ghatao. CPU/memory (ya custom metrics) dekh ke replicas adjust karta hai.

◆
"CPU 70% cross? ek aur Pod add karo, max 6 tak. Traffic gaya? wapas min 2 pe aa jao." Sab automatic, tum sote raho.
autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 6
  targetCPUUtilizationPercentage: 70  # 70% pe scale up
VANTA

Prod overlay mein optional hpa.yaml. Dev mein fixed 1 replica (learning ke liye scaling off).

BILLFREE

Har service default HPA on: min 2, max 6, 70% CPU. Real traffic pe api-gateway apne aap scale karta hai.

🚧
PDB — Pod Disruption Budget
kind: PodDisruptionBudget

Maintenance ke waqt (node drain, upgrade) Kubernetes Pods hataata hai. PDB kehti hai "kam se kam itne Pods hamesha zinda rehne chahiye" — sab ek saath mat hatao.

◆
Bina PDB ke node drain karo to saare replicas ek saath ja sakte hain → downtime. PDB minAvailable: 1 = "chahe kuch ho, 1 to chalta rahe."
VANTA

Optional pod-disruption-budgets component available hai — production-readiness dikhane ke liye.

BILLFREE

Har service chart mein pdb.enabled: true, minAvailable: 1. Node maintenance pe zero downtime guarantee.

🔒
SecurityContext
pod/container hardening

Container ko kam se kam power do — agar hack bhi ho to nuksaan seemit. Root mat chalao, filesystem read-only, extra permissions drop.

securityContext:
  runAsNonRoot: true          # root nahi
  readOnlyRootFilesystem: true # file system likha na jaaye
  allowPrivilegeEscalation: false
  capabilities: { drop: [ALL] }  # saari extra powers hatao
VANTA

frontend: runAsNonRoot, readOnlyRootFilesystem, drop: ALL, non-root user 1000. Google ka sample already hardened.

BILLFREE

Chart default sabke liye: non-root user 1000, read-only FS, all caps dropped. Scratch chahiye to emptyDir volume.

05

Access Control — kaun kya kar sakta hai

🪪
ServiceAccount & RBAC
identity + permissions

ServiceAccount = Pod ki identity (Pod ka "aadhaar card"). RBAC (Role-Based Access Control) = us identity ko kya karne ki permission hai.

  • ServiceAccount — har Pod ek SA ke naam pe chalta hai. Yeh batata hai "main kaun hoon".
  • Role / ClusterRole — permissions ka set ("Pods padh sakta hai, delete nahi").
  • RoleBinding — SA ko Role se jodta hai ("is identity ko yeh permissions do").
◆
ServiceAccount = tum kaun ho (ID card). Role = tum kya kar sakte ho (permissions). RoleBinding = ID card pe permissions chipka dena.
VANTA

Har service ka apna ServiceAccount (frontend, cartservice…). Least-privilege identity.

BILLFREE

ArgoCD ko cluster mein deploy karne ki RBAC permissions. Har service apni SA se chalti hai. Human access bhi RBAC se controlled.

06

Delivery — YAML cluster tak pahunchte kaise hain

Ab tak sab objects dekhe. Par 12 services ki YAML manually apply karna? Nahi. Yeh tools wahi automate karte hain — aur yahi dono projects ka sabse bada farak hai.

🧩
Kustomize
VANTA ka tareeka

Ek base (common YAML) banao, phir har environment ke liye overlay (sirf farak wali cheez). Copy-paste nahi — base + patch.

kustomize/ ├── base/ ← saari common YAML (frontend, cartservice, ...) └── overlays/ ├── dev/ ← replicas: 1, image: latest, NodePort 30080 ├── staging/ ← replicas: 2 └── prod/ ← replicas: 3, + HPA
# overlays/dev/kustomization.yaml
resources: [../../base]
replicas:
  - { name: frontend, count: 1 }   # base ko override
patches:                              # LB → NodePort (kind ke liye)
  - target: { kind: Service, name: frontend-external }
✓
Kustomize Kubernetes mein built-in hai — kubectl apply -k overlays/dev. Koi extra tool install nahi. Simple projects ke liye perfect.
⎈
Helm
Billfree ka tareeka

YAML ka template engine — "package manager for Kubernetes". Ek reusable chart banao (blueprint with blanks), har service bas apni values.yaml se blanks bhare.

◆
Billfree ki 7 services alag-alag YAML nahi likhi — ek microservice chart hai, har service sirf apni values deti hai (naam, image, replicas). DRY at scale.
deploy/charts/microservice/ ← ek reusable blueprint templates/ deployment.yaml service.yaml hpa.yaml pdb.yaml ingress.yaml servicemonitor.yaml deploy/apps/auth-service/values.yaml ← bas farak wali values deploy/apps/api-gateway/values.yaml ← bas farak wali values ... 7 services, 1 chart
ℹ
Kustomize vs Helm: Kustomize = base + patches (no variables, simple). Helm = full templating with variables/loops/conditions (powerful, thoda complex). Chhota project → Kustomize. Bahut services + reuse → Helm.
🔄
ArgoCD / GitOps
kind: Application

GitOps ka core: Git = single source of truth. Cluster ko manually kabhi haath nahi lagaate. ArgoCD cluster mein baithke Git ko watch karta hai, aur cluster ko Git jaisa bana deta hai.

◆
Tum sirf Git mein commit karte ho (image tag, replicas). ArgoCD dekh leta hai aur khud cluster update kar deta hai. Rollback? git revert — bas.
Developer → git push (config repo) │ ▼ [ ArgoCD in-cluster ] ← Git ko watch karta hai │ "Git bola 3 replicas, cluster mein 2? theek karo" ▼ Cluster = Git ke jaisa (auto-sync + self-heal)

2 killer features:

  • Auto-sync — Git badla, cluster khud update. Manual kubectl apply nahi.
  • Self-heal / drift correction — kisi ne manually cluster badla? ArgoCD wapas Git jaisa kar deta hai. Cluster kabhi Git se bahar nahi jaata.
VANTA

argocd/application.yaml — repo grvtech1/VANTA-Boutique ko watch karke Kustomize overlays apply karta hai.

BILLFREE

ApplicationSet per environment (dev/staging/prod) — ek pattern se saari services ki Apps auto-generate. root.yaml app-of-apps pattern. Ingress, monitoring, platform sab GitOps se.

✓
CI vs CD yaad rakho: CI (GitHub Actions) image banata + Git mein tag likhta hai. CD (ArgoCD) Git se padhta + cluster deploy karta hai. CI kabhi cluster ko touch nahi karta — yeh GitOps ki jaan hai.
07

Observability — cluster ke andar dikhta kya hai

📊
Prometheus & Grafana
metrics + dashboards

Prometheus har service se metrics kheenchta (scrape) hai — kitne requests, kitne errors, kitni latency. Grafana unhe sundar dashboards/graphs mein dikhata hai. Alert bhi bhejta hai.

  • ServiceMonitor — Prometheus ko batata hai "is service ko /metrics pe scrape karo".
  • PrometheusRule — alert rules ("error rate 5% cross 5 min → critical alert").
  • RED metrics — Rate (kitne req), Errors (kitne fail), Duration (kitna time). Har service ki sehat.
VANTA

Optional Prometheus/Grafana setup — learning ke liye. PLATFORM.md runbook mein documented.

BILLFREE
