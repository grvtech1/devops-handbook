Helm & Helm Chart

"Kubernetes ka package manager" — YAML ka template engine
🍪 Real-life misaal — cookie cutter

Socho tumhe 7 cookies banani hain — sab ek jaisi shape ki, bas naam alag (kisi pe "A", kisi pe "B"). Tum 7 baar haath se shape nahi banaoge. Ek cookie-cutter (साँचा) banao, aur har cookie pe alag naam daal do. साँचा same, decoration alag.

Helm chart = cookie cutter (साँचा, blanks ke saath). values.yaml = decoration (har cookie ka naam). helm install = cookie banana (asli deploy).

Billfree mein 7 microservices hain (api-gateway, auth-service, report-service…). Har ek ke liye alag-alag 100-line Deployment YAML likhna? Pagalpan. Ye sab 90% same hote hain — bas naam, image, port alag.

Isliye Helm ek template banata hai jismein "blanks" hote hain {{ .Values.naam }}. Har service sirf apni chhoti si values.yaml deti hai jo blanks bhar deti hai.

Chart template (साँचा)
kind: Deployment
metadata:
  name: {{ .Values.name }}
spec:
  replicas: {{ .Values.replicaCount }}
  # ... container ...
    image: {{ .Values.image }}
+
auth-service/values.yaml
name: auth-service
replicaCount: 2
image: .../auth:latest
=
helm install chalao → Helm blanks bhar ke poori asli YAML generate karta hai → cluster mein deploy. Ek साँचा, 7 services.
1
Chart banao (ek baar)
Blanks wali templates — deployment, service, hpa, pdb, ingress. Billfree: deploy/charts/microservice/
2
Har service apni values.yaml de
Sirf farak wali cheezein — naam, image, replicas. Billfree: deploy/apps/auth-service/values.yaml
3
Helm blanks bhar ke YAML banata hai
Template + values = complete Deployment/Service/HPA YAML
4
Cluster mein deploy
Saari generated YAML apply. 7 services, ek hi chart se.
TERMS CLEAR Chart = poora package (templates + default values). Release = jab tum chart ko install karte ho (ek chalti-phirti copy). values.yaml = blanks ki values. Ek chart se kai releases (dev/staging/prod) ban sakti hain.
💡
VANTA vs Billfree: VANTA Kustomize use karta hai (base + patch, no blanks — simple). Billfree Helm (blanks/templates — powerful). Dono ka kaam same: environment-wise YAML banana. Chhota project → Kustomize. Bahut services + reuse → Helm.

ConfigMap

App ki settings — image ke bahar rakhi hui
📱 Real-life misaal — phone ki settings

WhatsApp ek hi app hai (image). Par tumhare phone pe settings alag — dark mode on, language Hindi, notifications off. Mere phone pe settings alag. App wahi, settings alag.

Agar settings app ke andar hardcode hoti, to har change pe naya WhatsApp banana padta! Isliye settings app se bahar rakhi jaati hain. Wahi ConfigMap hai.

App ko chalne ke liye kuch settings chahiye — dusri service ka address, kaunsa environment, feature flags. Ye sab image ke andar hardcode karo to har chhoti setting badalne pe poori image dobara build karni padegi.

ConfigMap ye settings image ke bahar rakhta hai. Same image dev/staging/prod mein chalti hai — bas ConfigMap alag.

apiVersion: v1
kind: ConfigMap
metadata:
  name: frontend-config
data:                                # bas key-value settings
  CART_SERVICE_ADDR: "cartservice:7070"
  CURRENCY_SERVICE_ADDR: "currencyservice:7000"
  ENV_PLATFORM: "local"

Pod ye settings env variable banake uthata hai:

# Pod ke andar
envFrom:
  - configMapRef:
      name: frontend-config   # saari keys env var ban gayi
◆
Faayda: Settings badli? Bas ConfigMap update + Pod restart. Image rebuild nahi. "Config code se alag" — ye professional apps ka basic rule hai.
TUMHARE PROJECT MEIN VANTA ke frontend Deployment mein saare service addresses env: mein hain — CART_SERVICE_ADDR=cartservice:7070. Yahi ConfigMap ka kaam hai — settings alag, image se bahar.

Secret

ConfigMap jaisa hi — par sensitive cheezon ke liye
🔒 Real-life misaal — khuli shelf vs locker

Ghar mein kuch cheezein khuli shelf pe rakhte ho — books, remote (koi baat nahi kisi ne dekh li). Par paisa, gehne, passport? Locker mein. Kaam dono ka same (cheez rakhna), par ek locked.

ConfigMap = khuli shelf (normal settings). Secret = locker (password, API key, token). Bilkul same tarika, bas Secret protected hai.

Secret aur ConfigMap bhai-bhai hain — dono key-value data rakhte hain, dono Pod mein env var ban jaate hain. Farak: Secret sensitive data ke liye — DB password, API keys, tokens. Iska access RBAC se controlled hota hai.

apiVersion: v1
kind: Secret
metadata:
  name: postgres-secret
type: Opaque
data:
  POSTGRES_PASSWORD: c3VwZXJzZWNyZXQ=   # base64 (encryption NAHI)

Pod bilkul ConfigMap jaise uthata hai — bas secretRef:

envFrom:
  - secretRef:
      name: postgres-secret   # $POSTGRES_PASSWORD mil gaya
⚠
Sabse zaroori rule: Secret ki asli value kabhi Git mein commit mat karo. base64 koi security nahi — 1 second mein decode ho jaata. Git mein daala = history mein hamesha, sabko dikhega.
1
Secret banta hai — cluster setup ke waqt, ek baar
Script se, random password generate karke. kubectl create secret ... --from-literal=PASSWORD=$(openssl rand)
2
Cluster ke andar rehta hai (etcd mein)
Git mein nahi — sirf cluster ke andar. Git mein sirf .example.yaml (nakli placeholder).
3
Pod uthata hai secretRef se
Container ke andar env variable ban jaata — bina value dikhaye
TUMHARE BILLFREE MEIN (asli production pattern) postgres-secret Git mein nahi hai. bootstrap-cluster.sh random password se banata hai. StatefulSet envFrom: secretRef se uthata hai. ArgoCD ise manage nahi karta (kyunki Git mein hai hi nahi). Yahi sahi tareeka hai.

Init Container

Main container se pehle chalne wala helper
🍽️ Real-life misaal — restaurant kholne se pehle

Restaurant subah customers ke liye khulne se pehle — safai wala aata hai, tables lagti hain, kitchen ready hoti hai. Ye setup staff apna kaam karke chala jaata hai. Uske baad hi restaurant (main kaam) shuru hota hai.

Init container = setup staff. Pehle chalta hai, apna kaam poora karke khatam ho jaata hai, phir main container (restaurant) start hota hai.

Ek Pod mein main container chalne se pehle kuch taiyari karni ho — init container wahi karta hai. Rule: init container poora chal ke khatam ho, tabhi main container start hoga. Init fail hua? Main chalega hi nahi.

spec:
  initContainers:                # PEHLE ye chalega
    - name: wait-for-db
      image: busybox
      command: ['sh','-c','until nc -z postgres 5432; do sleep 2; done']

  containers:                    # db ready hone ke BAAD ye
    - name: app
      image: auth-service:latest
Pod start │ ▼ [ initContainer: wait-for-db ] ← postgres ready hone tak wait │ ✓ khatam ho gaya ▼ [ main container: app ] ← ab safely start

Kis kaam ke liye use hota hai:

A
Wait karna
DB / dusri service ready hone tak ruko — warna app crash ho jaayega
B
DB migration
App start se pehle database ki tables banao/update karo
C
Files download
Config / certificates pull karke shared volume mein rakho
◆
Init vs Sidecar mat confuse karna: Init container khatam ho jaata hai phir main chalta hai. Sidecar main ke saath-saath chalta rehta hai (jaise logging agent jo poore time logs bhejta hai).
TUMHARE BILLFREE MEIN migrate-job yaad hai? App deploy hone se pehle database migrations chalata hai. Yahi init-container/Job pattern hai — "main kaam se pehle taiyari".
