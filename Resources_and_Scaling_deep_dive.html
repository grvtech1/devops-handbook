Part A — Pod layer: requests & limits

Day 2/3 · Kubernetes

requests vs limits — do bilkul alag cheezein

Sabse zyada confuse hone wali jodi. Inka kaam ekdum alag hai.

requests = booking. "Mujhe kam se kam itna chahiye." Scheduler isi ko dekh ke decide karta hai ki Pod kis node pe jayega. Ye guarantee hai.

limits = ceiling. "Isse zyada mat lena." Runtime (kubelet/kernel) isko enforce karta hai. Ye hadd hai.

resources:
  requests: { cpu: 50m,  memory: 96Mi }   # booking — scheduler dekhta hai
  limits:   { cpu: 500m, memory: 256Mi }  # ceiling — kernel enforce karta hai
ℹ
Units: 1000m = 1 poora CPU core (m = millicore). 500m = aadha core. Mi = mebibyte (1024-based), M = megabyte (1000-based) — hamesha Mi use karo.

Restaurant analogy: requests = table reserve karna (guaranteed milegi). limits = "ek banda 3 plate se zyada nahi khayega" (hadd).

🔒 "Enforce" ka matlab kya — aur requests enforce hote hi nahi

Speed limit vs speed governor:

Sadak pe board "60 km/h"   →  sirf LIKHA hai. Tum 100 chala sakte ho.
                              koi PHYSICALLY nahi rok raha.

Gaadi mein speed governor  →  60 se upar jaa HI NAHI SAKTI.
                              accelerator dabate raho — kuch nahi hoga.
Enforce hota hai?Kaun dekhta hai
requests❌ NAHI — sirf ek numberScheduler — node choose karne ke liye
limits✅ HAAN — asli deewarKernel — runtime pe physically rokta hai

Matlab: requests: 100m likha aur app 500m kha raha hai, node pe CPU khaali hai? Koi nahi rokega. requests koi guard nahi — wo kaagaz pe booking hai. Par limits: 500m pe pahunchte hi — deewar.

Hota kaise hai — cgroups. Kubernetes jaadu nahi karta, wo Linux se kehta hai:

tumhara YAML: limits: {memory: 256Mi, cpu: 500m}
        ↓
kubelet container banata hai
        ↓
kubelet ye numbers cgroup files mein LIKH deta hai
        ↓
Linux kernel ab inhe PHYSICALLY enforce karta hai
# container ke andar ye files SACH mein hoti hain:
/sys/fs/cgroup/memory.max   → 268435456      # 256Mi bytes mein
/sys/fs/cgroup/cpu.max      → 50000 100000  # har 100ms mein sirf 50ms CPU
🐢 CPU — "quota khatam, ruk jao"
100ms period:
[■■■■■ chal raha 50ms][░░░░░ ROKA 50ms]
                          ↑ throttling

App marta nahi — jabardasti sulaa diya jaata hai agla period aane tak. Isliye latency badhti hai.

💀 Memory — "jagah nahi, mar jao"
app: "10MB aur chahiye" → malloc()
kernel: memory.max cross!
kernel: SIGKILL → exit 137

Memory ko throttle nahi kar sakte — jo byte chahiye wo chahiye. Kernel ke paas ek hi option: maar do.

🧪 LAB — apne limits ko kernel ke andar dekho (2 min)
kubectl exec -it <frontend-pod> -- sh

cat /sys/fs/cgroup/memory.max    # tumhara memory limit, bytes mein
cat /sys/fs/cgroup/cpu.max       # "quota period"
cat /sys/fs/cgroup/cpu.stat      # throttled_usec ← kitna roka gaya
✅ Wahi numbers milenge jo YAML mein likhe the — kubelet ne kernel ko bata diya. throttled_usec badh raha hai? CPU limit tight hai.

⚡ CPU aur Memory — limit cross karne pe alag anjaam

Ye ek baat 90% log nahi jaante. Interview mein direct poocha jaata hai.
ResourceNatureLimit cross kiya to?
CPUCompressible (dabaya ja sakta)🐢 Throttle — dheema kar diya. Pod zinda rehta hai, bas slow.
MemoryIncompressible (dabaya nahi ja sakta)💀 OOMKilled — Pod maar diya. Exit code 137.

Kyun? CPU time-share ho sakta hai — thodi der ruk jao, baad mein mil jayega. Memory nahi — jo byte chahiye wo chahiye. Nahi mila → kernel ka OOM-killer aake process maar deta hai.

CPU limit cross    →  CFS throttling  →  app slow  →  latency ↑  (zinda)
Memory limit cross →  OOM killer      →  SIGKILL   →  exit 137   (murda)
                                                        ↓
                                              CrashLoopBackOff
⚠
Ye tune khud dekha hai! VANTA pe emailservice/recommendationservice exit 137 de rahe the — liveness probe timeout 1s bahut aggressive tha, kubelet ne SIGKILL bhej diya. Memory ka usse koi lena-dena nahi tha. Neeche wala card isi ke liye hai.

💀 exit 137 ke DO baap hain

Ye sabse mehnga misdiagnosis hai. Zyadatar course (aur log) "137 = OOM" ratate hain — aur aadhe waqt galat fix karte hain.

137 = 128 + 9 = process ko SIGKILL mila. Par SIGKILL bhejne wale do hain — aur dono ka ilaaj ulta hai:

                        exit 137 = SIGKILL
                               │
              ┌────────────────┴────────────────┐
              ▼                                 ▼
   kernel ka OOM killer              kubelet
   "tune apni RAM limit todi"        "teri liveness probe fail hui"
              │                                 │
   Reason: OOMKilled                 Reason: Error
              │                                 │
   ✅ RAM limit badhao               ✅ probe relax karo
   ❌ probe chhuna bekaar            ❌ RAM badhana BEKAAR

Isliye ek hi rule hai — exit code se nahi, Reason: field se diagnose karo:

kubectl describe pod <name> | grep -A3 "Last State"

#   Reason: OOMKilled  → memory ka masla    → limits.memory badhao
#   Reason: Error      → probe/kubelet kill → probe relax karo
★
Direction yaad rakh — ek taraf sahi, dusri galat:
✅ "RAM limit toota → OOMKilled → 137" — ye sahi hai.
❌ "137 dikha → RAM limit badhao" — ye ulta hai aur galat. Har OOM 137 deta hai, par har 137 OOM nahi hota.

Do scenario — bilkul same symptom, ulta fix:

SCENARIO A — auth-service
Reason:  OOMKilled
Exit:    137
top:     254Mi / 256Mi  ← 99%

Diagnosis: sach mein RAM khatam. Fix: limits.memory 256Mi → 512Mi.
Phir poochho: memory badhi kyun? (leak?) — limit bump bandage hai, ilaaj nahi.

SCENARIO B — analytics-service
Reason:  Error
Exit:    137
top:     118Mi / 512Mi  ← 23% !!
Events:  Liveness probe failed

Diagnosis: RAM bilkul theek hai. Probe ne healthy pod maar diya. Fix: timeoutSeconds 1s → 5s.
Yahan limit badhaoge to kuch nahi hoga — pod phir marega.

ℹ
Dono mein exit 137 hai. Dono CrashLoopBackOff dikhte hain. Farak sirf ek shabd — Reason:. Isliye top hamesha chalao: memory 23% pe ho aur tum limit badha rahe ho, to tum galat ped pe bhonk rahe ho. 🐕
🧪 LAB — dono khud banao aur farak dekho (VANTA)
  1. OOM banao → kubectl set resources deploy/frontend --limits=memory=20Mi
  2. kubectl describe pod <pod> | grep -A3 "Last State" → Reason: OOMKilled ✓
  3. Wapas theek karo: --limits=memory=128Mi
  4. Ab probe-kill banao → liveness timeoutSeconds: 1, periodSeconds: 2 patch karo
  5. Wahi command → ab Reason: Error + Liveness probe failed events ✓
✅ Dono baar exit 137 — par Reason alag. Yahi ek field poora diagnosis hai.

🏅 QoS Classes — node bhar gaya to kaun pehle marega

Node pe memory khatam. Kubernetes kisi ko to nikaalna hi padega. Kise? Ye QoS decide karta hai.

Tum QoS class likhte nahi — Kubernetes tumhare requests/limits dekh ke khud assign karta hai.

QoS ClassKab milti haiEviction (nikalne) mein
Guaranteedrequests == limits (CPU + memory dono, har container)Sabse aakhir mein — sabse safe 🛡️
Burstablerequests set hain, par limits se kamBeech mein
BestEffortKuch bhi set nahi 😬Sabse pehle nikala jayega ☠️
 
# Guaranteed — requests == limits
resources:
  requests: { cpu: 500m, memory: 512Mi }
  limits:   { cpu: 500m, memory: 512Mi }   # ← bilkul same

# Burstable — requests < limits (zyadatar apps)
resources:
  requests: { cpu: 50m,  memory: 96Mi }
  limits:   { cpu: 500m, memory: 256Mi }

# BestEffort — kuch nahi (KABHI production mein nahi!)
# resources: {}
★
Senior call: Database / critical service → Guaranteed (requests == limits) taaki wo kabhi evict na ho. Normal web app → Burstable theek hai. BestEffort kabhi nahi — wo pehla shikaar hai.

⏳ Pod "Pending" atka hai — requests ki wajah se

Pod ban gaya par chal nahi raha, status Pending. Kyun?

Scheduler requests dekh ke node dhundhta hai. Agar kisi bhi node pe utni free (allocatable) capacity nahi — Pod hamesha ke liye Pending.

ℹ
Zaroori: Node ki poori RAM tumhe nahi milti!
Allocatable = Capacity − kube-reserved − system-reserved − eviction-threshold
Ek 4GB node pe shayad ~3.2GB hi pods ke liye ho. Isliye "4GB node pe 4GB ka pod" kabhi schedule nahi hoga.
kubectl describe pod <name>
# Events:
#   Warning  FailedScheduling  0/3 nodes are available:
#   3 Insufficient cpu, 3 Insufficient memory.   ← yahi clue

kubectl describe node <node>    # Allocatable vs Allocated dekho
kubectl top nodes                # actual usage

Fix ke 2 raaste: (1) Pod ke requests kam karo — agar wo zyada maang raha tha. (2) Node layer pe jao — aur nodes lao ya bada instance (Part B).

🎯 Scheduler kaam kaise karta hai — Filter → Score

Har Pending pod do phase se guzarta hai. Ye jaanna zaroori hai kyunki Pending ka matlab hai tum phase ① mein hi bahar ho gaye:

Saare nodes ──▶ ① FILTER ──▶ jo bache ──▶ ② SCORE ──▶ winner
                "yahan CHAL SAKTA hai?"          "inme se BEST kaun?"
                hard pass/fail                    ranking 0–10

① Filter — koi bhi ek fail, node bahar. Har check ka alag message hai — aur alag ilaaj:

CheckMessage jo dikhegaIlaaj
Allocatable kaafi hai?Insufficient cpu / memoryrequests kam karo ya node lao
Taint tolerate karta hai?untolerated taint {...}toleration do (jagah hai, permission nahi)
nodeSelector/affinity match?didn't match node affinity/selectorlabel/affinity theek karo
PV is zone se reachable?volume node affinity conflictTerraform — us AZ mein node lao

② Score — sirf survivors compete karte hain. Emptier node prefer hota hai, aur PodTopologySpread replicas ko alag nodes pe faila deta hai — isliye tere 3 replicas apne aap 3 nodes pe jaate hain, tune kuch nahi kaha.

★
Filter = eligibility (degree hai? experience hai? nahi to bahar). Score = interview ranking (jo bache unme best kaun).

Pending ka matlab: tum Filter mein hi bahar ho — Score tak pahunche hi nahi. Isliye "node khaali hai phir bhi Pending" bilkul mumkin hai — jagah thi, par kisi aur check ne roka. Exact rejection string padho, andaaza mat lagao.
ℹ
Ek line mein teen alag "Pending": Insufficient memory = capacity ka masla · untolerated taint = permission ka masla · volume node affinity conflict = zone ka masla (ye Kubernetes se fix hi nahi hoga — Terraform chahiye). Teeno "Pending" dikhte hain. Teeno ka ilaaj alag hai.

❌ Pod layer ki 4 classic galtiyan — aur sahi tareeka

❌ GALAT — kuch set hi nahi kiya
# resources: {}
# "chalega to sahi"

Anjaam: BestEffort QoS. Node pe pressure aaya → tumhara pod sabse pehle mara. Scheduler ko bhi nahi pata kitni jagah chahiye → node overcommit.

✅ SAHI — hamesha requests set karo
requests: { cpu: 50m, memory: 96Mi }
limits:   { memory: 256Mi }

Requests hamesha — bina iske scheduler andha hai.

❌ GALAT — memory limit >> requests
requests: { memory: 96Mi }
limits:   { memory: 2Gi }  # bada gap

Anjaam: Node pe 96Mi book hui, par app 2Gi tak kha sakta hai → node overcommit → doosre pods evict, ya node hi NotReady.

✅ SAHI — memory limit ≈ requests
requests: { memory: 256Mi }
limits:   { memory: 256Mi }

Memory ke liye gap chhota rakho. Predictable — aur Guaranteed QoS mil jaati hai.

❌ GALAT — CPU limit bahut tight
requests: { cpu: 100m }
limits:   { cpu: 100m }

Anjaam: Node pe CPU khaali pada hai phir bhi app throttle ho raha → latency spike. Sabse chhupa hua performance bug.

✅ SAHI — CPU ko saans lene do
requests: { cpu: 100m }
limits:   { cpu: 500m }
# ya CPU limit hi mat do

CPU throttle karta hai, marta nahi — isliye limit dheeli (ya na) rakho. Requests zaroori, CPU limit optional.

❌ GALAT — value andaaze se

"1 CPU / 1Gi de dete hain, safe rahega." → 10 pods × 1 CPU = 10 CPU book, actual use 0.5 CPU. Node bhara dikh raha, khaali hai. Paisa barbaad + Pending pods.

✅ SAHI — naapo, phir set karo
kubectl top pods
# ya VPA "Off" mode se
# recommendation lo

Actual usage dekho → requests ≈ p95 usage rakho → Git mein commit.

🧪 LAB 1 — OOMKilled khud banao (VANTA, kind)
  1. frontend ka memory limit bahut kam kar do: 
    kubectl set resources deploy/frontend --limits=memory=20Mi
  2. Watch karo — pod marega: 
    kubectl get pods -w
kubectl describe pod <frontend-pod> | grep -A3 "Last State"
  3. Exit code dekho — 137 aur reason OOMKilled milega
  4. Fix karo: 
    kubectl set resources deploy/frontend --limits=memory=128Mi
✅ Dikhna chahiye: Last State: Terminated · Reason: OOMKilled · Exit Code: 137
🧪 LAB 2 — Pending pod banao (requests bahut zyada)
  1. Itna CPU maango jo node pe hai hi nahi: 
    kubectl set resources deploy/frontend --requests=cpu=50
  2. kubectl get pods            # Pending dikhega
kubectl describe pod <pod>  # "Insufficient cpu"
  3. Node ki asli capacity dekho: 
    kubectl describe node | grep -A6 "Allocatable"
  4. Wapas theek karo: --requests=cpu=100m
✅ Samajh aana chahiye: Allocatable < Capacity (kube/system reserved nikal ke)
🧪 LAB 3 — QoS class check karo
kubectl get pod <pod> -o jsonpath='{.status.qosClass}'
# Guaranteed | Burstable | BestEffort

# saare pods ki QoS ek saath:
kubectl get pods -o custom-columns=NAME:.metadata.name,QOS:.status.qosClass
✅ VANTA ke pods zyadatar Burstable milenge (requests < limits)

Terraform ka ilaaka — machine, pod nahi

────────────────────────────┼──────────────────────────
EC2 instance (size, count)  │  Pod

BILLFREETera infra/terraform/compute.tf EC2 banata hai, cloud-init se kubeadm chalta hai → cluster ban jaata hai. Uske baad Terraform ka kaam khatam — aage Helm/ArgoCD.

💣 "forces replacement" — Day 7 ka sabse bada khatra

instance_type badla, apply kiya — aur production gir gaya. Kyun?

EC2 ka instance_type in-place badalta nahi (AWS mein stop→resize→start hota hai; Terraform default mein replace karta hai). To Terraform bolega:

terraform plan

# aws_instance.worker must be replaced
-/+ resource "aws_instance" "worker" {
      ~ instance_type = "t3.medium" -> "t3.xlarge" # forces replacement
    }
Plan: 1 to add, 0 to change, 1 to destroy.

1 to destroy = wo node mar jayega, uske saare Pods ke saath — bina kisi warning ke, bina drain ke.

❌ GALAT — seedha apply
terraform apply -auto-approve

Anjaam: Node instantly destroy → us pe chal rahe saare pods ek saath gaye → agar wahi replicas the to downtime. PVC attached tha to volume detach issues.

✅ SAHI — cordon → drain → apply
kubectl cordon <node>
kubectl drain <node> \
  --ignore-daemonsets \
  --delete-emptydir-data
terraform apply
kubectl uncordon <new-node>

Pods pehle shift → phir machine replace. Zero downtime.

★
PDB yahi bachata hai. drain PDB respect karta hai — minAvailable: 1 hai to ek saath sab pods nahi hatega, ek-ek karke shift honge. Tere Billfree chart mein PDB on hai — isliye drain safe hai. Bina PDB ke drain = sab replicas ek saath ja sakte hain.

❌ Terraform ki 4 classic galtiyan

GaltiKya hota haiSahi tareeka
plan skip karna Pata hi nahi chala ki 1 node destroy ho raha terraform plan hamesha — "1 to destroy" dhoondo
-auto-approve prod pe Confirm ka mauka nahi, galti seedha live Prod mein manual approve, ya CI mein plan → review → apply
tfstate local rakhna Laptop gaya → state gaya → Terraform ko infra "dikhna" band Remote backend (S3 + DynamoDB lock) — team safe
Console se manually badalna Drift — Git kuch aur kehta, reality kuch aur. Agla apply usko ulta kar dega Sab change sirf Terraform se. Drift check: terraform plan
⚠
tfstate = Terraform ki yaaddasht. Usme likha hai "maine kya-kya banaya". Delete ho gaya → Terraform samjhega kuch bana hi nahi → dobara sab banayega (duplicate infra!) ya orphan chhod dega. Isliye S3 mein, versioning on, DynamoDB lock ke saath.

🗣️ Interview mein aise bolo

"Pehle main poochta hoon bottleneck kis layer pe hai. Pod OOMKilled ho raha hai (exit 137) to ye pod-level hai — resources.limits Git mein badhaunga, ArgoCD sync kar dega. Agar pods Pending hain to node capacity ka masla hai — main describe node se Allocatable dekhunga, aur horizontal jaunga (aur nodes) kyunki vertical ki ceiling hoti hai. Agar instance_type badalna hi pade to Terraform se — par pehle terraform plan mein 'forces replacement' check karunga, aur cordon + drain karunga taaki pods safely shift ho jayein; PDB unhe ek saath girne se rokega.

Aur main requests hamesha set karta hoon — bina uske scheduler andha hai aur pod BestEffort QoS mein chala jaata hai, jo pressure pe sabse pehle evict hota hai. Memory ka limit main requests ke kareeb rakhta hoon (predictable, Guaranteed QoS), par CPU limit dheela — kyunki CPU throttle karta hai, marta nahi."
